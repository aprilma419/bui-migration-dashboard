RHDH 2.0: MUI to BUI Transition Work PlanPhase 1: Audit & Inventory (Week 1)Goal: Establish the baseline of what we use versus what exists in the new ecosystem.Task 1.1: Plugin Component Extraction. Run a static analysis (e.g., using grep or custom scripts) across RHDH plugins to identify all @mui/material and @mui/lab components currently in production.Task 1.2: BUI Cataloging. Review the Backstage UI components to document available components, their stability status (Alpha/Beta/Stable), and their underlying architecture.Task 1.3: The Gap Analysis. Create a master spreadsheet comparing MUI usage to BUI availability.Direct Match: MUI Button $\rightarrow$ BUI Button.Partial Match: Component exists but lacks specific props required by RHDH.Missing: Component exists in MUI but has no BUI equivalent (e.g., specialized DataGrids or Pickers).Phase 2: Design Token & PatternFly Mapping (Week 2)Goal: Ensure visual consistency by bridging the gap between BUI's token system and PatternFly’s variables.Task 2.1: Token Extraction. Document BUI design tokens for colors, typography, spacing, and elevation.Task 2.2: PatternFly Alignment Matrix. Map BUI tokens to the corresponding PatternFly CSS variables.Example: Mapping BUI color.background.canvas to PatternFly --pf-v5-global--BackgroundColor--100.Task 2.3: Theming Prototype. Create a small POC (Proof of Concept) where a BUI component is wrapped in a theme provider that injects PatternFly variable values into BUI tokens.Phase 3: Strategy & "Parallel Life" Architecture (Week 3)Goal: Solve the technical challenge of running MUI and BUI side-by-side.Task 3.1: CSS Namespace & Isolation. Define the strategy to prevent CSS collisions.Option A: Use MUI’s ClassNameGenerator to prefix MUI classes.Option B: Shadow DOM encapsulation for specific plugin views.Task 3.2: Implementation Roadmap. Define the "Migration Tier" for components:Tier 1: Low-hanging fruit (Buttons, Typography, Layout).Tier 2: Complex components (Tables, Dialogs).Tier 3: Custom RHDH components requiring upstream BUI contributions.Task 3.3: Blocker Identification. Finalize the list of "Critical Gaps" to be communicated to the upstream Backstage community or prioritized for internal development.Phase 4: Finalization & Team Alignment (Week 4)Goal: Document findings and get stakeholder buy-in.Task 4.1: Documentation. Compile the "RHDH 2.0 Migration Manifesto" covering the mapping matrix and the coexistence strategy.Task 4.2: Presentation. Host a technical deep-dive with the RHDH engineering team to review the implementation strategy.Task 4.3: Handover. Create Jira tickets for the actual migration work based on the Tiered approach defined in Phase 3.
